  Informatics Apartment Society - Implementation Plan

     Context

     Build a web-based housing society management system inspired by NoBroker Hood. The app handles financial management (income/expenses, maintenance billing, ledgers, balance sheets), member
     management (manual + Excel upload), custom role-based access control, and gate security management (visitor entry/exit, gatekeeper approval workflow). This is a greenfield project starting
     from an empty directory.

     Tech Stack

     Core Modules (6 total)

     1. Auth & RBAC - Firebase login, custom roles with granular permissions
     2. Member Management - Residents CRUD, Excel upload, unit assignment
     3. Financial Management - Income/expense, maintenance billing, ledger, balance sheet
     4. Gate Management - Visitor logging, entry requests, gatekeeper panel
     5. Notices & Complaints - Society announcements, ticketing system
     6. Dashboard - Key metrics, charts, quick actions

     ---
     Implementation Phases

     Phase 1: Project Bootstrap & Infrastructure

     Files created:
     - package.json, next.config.ts, tsconfig.json, tailwind.config.ts
     - prisma/schema.prisma - Full database schema
     - src/middleware.ts - Auth + route protection
     - src/lib/db/prisma.ts - Prisma client singleton
     - src/lib/auth/ - Firebase client, admin, session management
     - src/components/providers/ - App providers (auth, query, theme, toast)
     - .env.example

     Steps:
     1. npx create-next-app@latest with TypeScript, Tailwind, App Router
     2. Install all dependencies (shadcn/ui, Prisma, Firebase, TanStack Query, etc.)
     3. Initialize shadcn/ui and add core components (button, card, input, table, dialog, form, etc.)
     4. Set up Prisma with Neon adapter and write full schema
     5. Configure Firebase auth (client + admin SDK wrappers)
     6. Build auth middleware with next-firebase-auth-edge

     Database Models (Prisma):
     - Society - name, address, logo, registrationNo
     - User - firebaseUid, email, name, phone, societyId
     - Role - name, description, permissions (JSON), societyId, isSystem
     - UserRole - userId, roleId (many-to-many)
     - Block - name, societyId
     - Unit - block, floor, unitNumber, type, area, societyId
     - Member - userId, unitId, ownershipType (OWNER/TENANT), moveInDate, isActive
     - Transaction - type (INCOME/EXPENSE), category, amount, date, description, createdById
     - BillingCycle - name, month, year, dueDate, status
     - MaintenanceBill - billingCycleId, memberId, unitId, totalAmount, status
     - BillLineItem - billId, head, amount
     - Payment - billId, amount, paymentDate, method, transactionRef
     - Visitor - name, phone, purpose, vehicleNo, entryTime, exitTime, approvedById
     - EntryRequest - visitorName, purpose, unitId, status (PENDING/APPROVED/REJECTED)
     - PreApproval - visitorName, visitorPhone, validFrom, validUntil, memberId
     - Notice - title, content, category, priority, authorId
     - Complaint - title, description, category, status, priority, filedById, assignedToId
     - Comment - complaintId, content, authorId
     - AuditLog - action, entityType, entityId, userId, metadata

     Every tenant-scoped table has societyId for multi-tenancy.

     Phase 2: Auth & App Shell

     Files:
     - src/app/(auth)/layout.tsx - Centered auth layout
     - src/app/(auth)/login/page.tsx - Email/password + Google sign-in
     - src/app/(auth)/register/page.tsx - Society admin registration
     - src/app/(dashboard)/layout.tsx - Sidebar + header + auth guard
     - src/components/layout/app-sidebar.tsx - Navigation sidebar
     - src/components/layout/app-header.tsx - Top header with user menu
     - src/components/shared/permission-gate.tsx - Conditional render by permission
     - src/lib/auth/rbac.ts - Permission checking utilities
     - src/hooks/use-auth.ts - Auth state hook
     - src/hooks/use-permission.ts - Permission check hook

     Auth flow: Firebase handles email/password + Google login -> next-firebase-auth-edge middleware validates tokens via cookies -> server components read user from cookie -> RBAC checks
     permissions at middleware, API, and UI layers.

     Phase 3: Role-Based Access Control

     Files:
     - src/app/(dashboard)/roles/page.tsx - Roles list
     - src/app/(dashboard)/roles/new/page.tsx - Create role
     - src/app/(dashboard)/roles/[roleId]/edit/page.tsx - Edit role permissions
     - src/app/api/roles/route.ts - CRUD API
     - src/services/role.service.ts - Business logic
     - src/lib/constants/permissions.ts - Permission definitions

     Permission system: Roles store a JSON array of permission strings (e.g., ["MEMBER_READ", "MEMBER_WRITE", "FINANCE_READ"]). Pre-seeded system roles: Super Admin, Secretary, Treasurer,
     Committee Member, Resident, Gatekeeper. Societies can create unlimited custom roles.

     Permission categories:
     - MEMBER_READ, MEMBER_WRITE, MEMBER_DELETE
     - FINANCE_READ, FINANCE_WRITE
     - MAINTENANCE_READ, MAINTENANCE_WRITE, MAINTENANCE_GENERATE
     - GATE_READ, GATE_WRITE, GATE_APPROVE
     - NOTICE_READ, NOTICE_WRITE
     - COMPLAINT_READ, COMPLAINT_WRITE, COMPLAINT_ASSIGN
     - ROLE_MANAGE, SETTINGS_MANAGE
     - GATEKEEPER (special role for gate panel access)

     Phase 4: Member Management

     Files:
     - src/app/(dashboard)/members/page.tsx - Member list (search, filter, paginate)
     - src/app/(dashboard)/members/new/page.tsx - Add member form
     - src/app/(dashboard)/members/upload/page.tsx - Excel bulk upload
     - src/app/(dashboard)/members/[memberId]/page.tsx - Member detail
     - src/app/(dashboard)/members/[memberId]/ledger/page.tsx - Individual ledger
     - src/app/api/members/route.ts - CRUD API
     - src/app/api/members/upload/route.ts - Excel upload API
     - src/services/member.service.ts - Business logic
     - src/services/excel.service.ts - Excel parsing
     - src/lib/validations/member.schema.ts - Zod validation
     - src/components/shared/data-table/ - Reusable data table components

     Excel upload: Uses xlsx library to parse uploaded .xlsx/.csv files. Shows preview of parsed data with validation errors before confirming import. Maps columns to member fields (name, email,
     phone, unit, ownership type).

     Phase 5: Financial Management

     Files:
     - src/app/(dashboard)/finance/page.tsx - Finance overview with charts
     - src/app/(dashboard)/finance/income/ - Income CRUD pages
     - src/app/(dashboard)/finance/expenses/ - Expense CRUD pages
     - src/app/(dashboard)/finance/maintenance/ - Billing cycle management
     - src/app/(dashboard)/finance/maintenance/generate/page.tsx - Generate bills
     - src/app/(dashboard)/finance/balance-sheet/page.tsx - Balance sheet report
     - src/app/(dashboard)/finance/receipts/page.tsx - Payment receipts
     - src/app/api/finance/ - All finance API routes
     - src/services/finance.service.ts - Transaction logic
     - src/services/maintenance.service.ts - Bill generation & payment reconciliation

     Maintenance billing flow: Admin creates a billing cycle (month/year) -> defines billing heads (maintenance, water, parking, sinking fund, etc.) with amounts -> generates bills for all active
     members -> members see pending bills -> payments recorded against bills -> ledger updated automatically.

     Balance sheet: Aggregates all income transactions, expense transactions, pending receivables, and liabilities to generate a standard balance sheet view with date range filtering.

     Phase 6: Gate Management

     Files:
     - src/app/(dashboard)/gate/page.tsx - Gate management overview
     - src/app/(dashboard)/gate/visitors/page.tsx - Visitor log
     - src/app/(dashboard)/gate/entry-requests/page.tsx - Pending approvals
     - src/app/(dashboard)/gate/pre-approvals/ - Pre-approval management
     - src/app/gatekeeper/layout.tsx - Standalone gatekeeper panel layout
     - src/app/gatekeeper/page.tsx - Quick visitor entry form
     - src/app/gatekeeper/log/page.tsx - Today's visitor log
     - src/app/api/gate/ - Gate API routes
     - src/services/gate.service.ts - Visitor management logic

     Gate flow: Gatekeeper logs visitor at gate -> system sends notification to resident -> resident approves/denies from their dashboard -> gatekeeper sees result. Residents can also pre-approve
     expected visitors with a time window.

     Gatekeeper panel: Separate route (/gatekeeper) with minimal tablet-friendly UI. Only accessible to users with GATEKEEPER role. Shows quick entry form, today's log, and pending approvals.

     Phase 7: Notices & Complaints

     Files:
     - src/app/(dashboard)/notices/ - Notice CRUD pages
     - src/app/(dashboard)/complaints/ - Complaint CRUD pages
     - src/app/api/notices/ & src/app/api/complaints/ - API routes
     - src/services/notice.service.ts & src/services/complaint.service.ts

     Notices: Categories (General, Maintenance, Event, Emergency). Priority levels. Created by committee members, visible to all residents.

     Complaints: Filed by residents, assigned to responsible roles. Status flow: Open -> In Progress -> Resolved -> Closed. Comment thread for updates.

     Phase 8: Dashboard & Settings

     Files:
     - src/app/(dashboard)/dashboard/page.tsx - Main dashboard
     - src/app/(dashboard)/settings/ - Society settings pages
     - src/app/(dashboard)/account/page.tsx - User account

     Dashboard widgets:
     - Total members count & occupancy rate
     - Monthly income vs expenses chart (Recharts)
     - Pending maintenance dues summary
     - Recent visitor activity
     - Open complaints count
     - Recent notices

     Settings: Society profile (name, address, logo), billing configuration (heads, amounts), unit/block management, society registration details.

     7. Dashboard: Verify all widget data matches underlying records
